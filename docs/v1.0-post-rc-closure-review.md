# BLUE-TANUKI v1.0 Post-RC Closure Review

This review closes the implementation-adjacent residual items identified after
the v1.0 release-candidate preparation.

## Bundle Integrity

Release integrity is defined by the generated sidecars:

- `release/blue-tanuki-1.0.0-rc.1-source-bundle.sha256`
- `release/blue-tanuki-1.0.0-rc.1-source-bundle.manifest.json`

The RC document intentionally does not embed a literal bundle SHA. The RC
document is included in the source bundle, so changing a SHA inside that
document changes the archive hash again. The stable validation path is:

```bash
pnpm release:bundle
pnpm release:verify
```

`release:verify` checks the archive, sidecar checksum, manifest, required files,
and forbidden secret-like entries.

## Preview Channel Promotion Decision

Slack, Discord, Teams, and LINE remain `first-party-preview`.

Promotion is blocked for v1.0 RC because:

- owner-run credentialed live smoke has not completed
- owner recovery steps for token revocation, target errors, permissions, and
  rate limits have not been rehearsed
- Teams and LINE still lack release-complete gateway-owned inbound webhook or
  subscription listeners
- preview promotion would expand permanent support burden beyond the verified
  owner operation surface

No compatibility-matrix promotion is made by this review.

## Channel Promotion Gate

Phase 11-S11 adds a machine gate for future first-party promotion:

```bash
pnpm validate:channels
```

Status: PASS with the current matrix.

The gate rejects Slack / Discord / Teams / LINE promotion unless redacted
owner-run live-smoke evidence and recovery-review evidence are provided. Teams
and LINE also require gateway-owned inbound listener closure before promotion.
This keeps the RC support boundary explicit instead of treating optional token
absence as a hidden pass.

## Plugin Review Gate

Phase 11-S12 adds the machine gate for Layer B plugin / skill / adapter submissions:

```bash
pnpm plugin:review -- --package <plugin-package-dir>
```

Status: PASS for the bundled workspace review path.

Submission mode requires `blue-tanuki.review.json` evidence and rejects wildcard
capabilities, manifest/package drift, lifecycle install scripts, runtime
dynamic import, forbidden WhatsApp-specific routes, final-review bypass claims,
and `kind=core` Layer B submissions.

The Plugin Review Gate result is non-authority evidence. It cannot approve,
execute, classify risk, bypass HDS-BRAIN or Approval Gate, or promote preview
status.

## Signed Installer Decision

No signed native installer is implemented by this review.

Signing requires a separate distribution security phase covering:

- code-signing certificate ownership
- platform-specific signing and notarization
- key storage and rotation
- release provenance
- installer tamper verification
- rollback behavior for failed signed installs

Until that exists, v1.0 RC remains an unsigned source-bundle distribution with
portable install scripts.

## Automatic Updater Decision

No automatic updater is implemented by this review.

An updater changes installed code and service metadata. In BLUE-TANUKI terms it
is a privileged update operation and must have its own security phase before it
can ship.

Minimum requirements before implementation:

- signed update manifest
- archive checksum verification
- rollback to previous verified version
- config and local data preservation
- audit record of update decision and result
- explicit owner approval for update execution
- failure behavior that does not silently alter authority policy

v1.0 RC continues to use operator-run source or release-bundle updates through
`docs/UPDATE_ROLLBACK_RUNBOOK.md`.
