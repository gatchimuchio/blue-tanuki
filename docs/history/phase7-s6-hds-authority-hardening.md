# Phase 7-S6: HDS Authority / Process / Memory Hardening

## Objective

Tighten the BLUE-TANUKI differentiation layer: HDS-BRAIN is not a chat wrapper, and not a loose tool router. It is the upstream authority OS. This phase closes the main gaps left after S5:

1. Actor/process metadata cannot be used as an authority escalation path.
2. `actor_policy` is enforced before any downstream command leaves HDS-BRAIN.
3. `execution_policy` is enforced before any downstream command leaves HDS-BRAIN.
4. HDS long-term memory records actor/process/commit snapshots, but remains non-authority.
5. Runtime snapshot exposes machine-readable invariants for the Control Center.

## Safety invariants

The following invariants are now represented in code and exposed in `HDSRuntimeSnapshot.invariants`:

```json
{
  "hds_calls_llm": false,
  "process_policy_enforced": true,
  "external_metadata_can_escalate_authority": false,
  "memory_used_for_authority": false,
  "final_review_boundary_enforced_by_approval_gate": true
}
```

## External metadata rule

External channel metadata is untrusted by default.

The following fields are ignored unless metadata contains:

```json
{
  "blue_tanuki.authority_context": "gateway_internal_v1"
}
```

Affected fields:

- `blue_tanuki.actor_kind`
- `actor_kind`
- `blue_tanuki.trust_level`
- `trust_level`
- `blue_tanuki.process_kind`
- `process_kind`
- `trigger_kind`
- `blue_tanuki.resume`
- `resume`

This prevents a channel/user/webhook payload from promoting itself to `owner`, `approval.process`, or `cron.process`.

## Process policy enforcement

After Frame/Model/Commit, HDS-BRAIN now applies a hard process guard before emitting a command.

Checks:

- actor kind must be allowed by `process.actor_policy.allowed_actor_kinds`
- owner-required processes require `actor_kind === "owner"`
- webhook trigger requires webhook actor
- cron trigger requires cron/system/owner actor

Violation result:

```text
FAIL / process_authority_denied:<reason>
```

## Execution policy enforcement

If Commit returns `ASSERT`, HDS-BRAIN builds a candidate command, then validates it against `process.execution_policy` before it can enter `inflight`.

Checks:

- command type is allowed
- tool name is allowed for `tool_call`
- requested capabilities are within process envelope
- command timeout does not exceed process timeout

Violation result:

```text
FAIL / process_execution_policy_denied:<reason>
```

This is the key guard that prevents, for example, a `webhook.process` from falling through to a default `llm_call`.

## Memory hardening

Long-term memory entries now include optional structural snapshots:

- `actor`
- `process`
- `commit`
- `tags`

These fields are copied from HDS logs. They are not generated by an LLM.

Memory retrieval remains deterministic:

- exact request id
- tag match
- recent entries

Memory remains explicitly non-authority:

```ts
memory_trace.used_for_authority === false
```

## Compatibility

The memory codec accepts legacy JSONL entries without the new optional fields. Hash canonicalization omits absent optional fields, so older memory chains continue to verify.

## Validation added

Tests added/updated:

- external metadata cannot upgrade actor authority
- gateway-internal authority context can set actor/process metadata
- webhook process cannot emit default LLM command
- runtime invariants are exposed
- long-term memory captures actor/process/commit/tags
- actor/process lookup works deterministically

## Design boundary

This phase does not add semantic memory, vector search, or LLM summarization.

Reason: those are useful for UX later, but they must remain below the authority plane. v0.1 keeps memory deterministic and auditable.
